*conduit.txt*  Remote file editing and SSH integration for Vim 9

                                 CONDUIT.VIM                                 *conduit*

Conduit.vim provides seamless integration between your local Vim instance and
remote SSH sessions. It allows you to open files from a remote shell directly
in your local Vim, and perform background file transfers (get/put) with
progress notifications.

NOTE: This plugin requires Vim 9.1 or later with +job, +popupwin, and
+reltime.

==============================================================================
1. Usage                                                    *conduit-usage*

To start using conduit, use the |:Conduit open| command to connect to a remote
host:
>
    :Conduit open user@example.com
<
This will open a terminal window running an SSH session. Within this session,
a special `lvim` command is available.

In the remote shell:
- `lvim file.txt` opens "file.txt" in your local Vim using netrw.
- `lvim split file.txt` opens it in a horizontal split.
- `lvim put file.txt` uploads a local "file.txt" to your remote CWD.
- `lvim get local_file.txt` downloads a remote file.

If `g:conduit_overwrite_vim` is set (default), `vim` is aliased to `lvim`. Use
`_vim` to run the actual vim binary on the remote host.

2. `:Conduit` Command                                      *conduit-commands*

:Conduit {subcommand} ...                                   *:Conduit*
        General command interface for Conduit operations.

        Subcommands:

:Conduit open [+SHORTOPT ...] [++LONGOPT ...] [++] [user@]host[:port]
                                                            *:Conduit-open*
        Open a new conduit session to {host}. Options before the host are
        written in short form (`+X`) or long form (`++name`). The option's
        identity, rather than its spelling, determines whether it configures
        ssh or Vim's terminal. Value-taking options accept either `=value` or
        a space-separated value.

        SSH options have both forms:
        `+B` / `++bindinterface`, `+b` / `++bindaddress`,
        `+c` / `++cipher`, `+D` / `++dynamicforward`,
        `+E` / `++logfile`, `+F` / `++config`, `+I` / `++pkcs11`,
        `+i` / `++identity`, `+J` / `++jump`, `+l` / `++login`,
        `+L` / `++localforward`, `+m` / `++mac`, `+O` / `++control`,
        `+o` / `++option`, `+p` / `++port`, `+Q` / `++query`,
        `+R` / `++remoteforward`, `+S` / `++controlpath`,
        `+W` / `++stdioforward`, and `+w` / `++tunnel`.
        For example, `+p 2222` and `++port 2222` are equivalent, as are
        `+J user@bastion` and `++jump user@bastion`. See
        |conduit-forwarding| for how forwarding options are scoped.

        Vim terminal options currently use long form: `++curwin`,
        `++vertical`, `++hidden`, `++norestore`, `++open`, `++close`,
        `++noclose`, `++shell`, `++rows=N`, `++cols=N`, `++eof=...`,
        `++api=...`, `++kill=...`, and `++opencmd=...`.

        A short-form name must be exactly one character; a long-form name
        must not be. Unlisted short-form names such as `+v` pass through raw
        to ssh, while long-form names are a closed, validated set. A bare
        `++` (or `--`) by itself stops option parsing before the host.

:Conduit deploy [+SHORTOPT ...] [++LONGOPT ...] [++] [user@]host[:port]
                                                            *:Conduit-deploy*
        Deploy the conduit environment to {host} without opening a terminal.
        This is useful when you already have an SSH session elsewhere and
        only want to inject the remote `lvim` environment. On success, Conduit
        echoes and copies the remote `source` command. Like |:Conduit-open|,
        this still opens its own background SSH session to carry the
        `lvim` reverse tunnel (and any forwarding option you request) — see
        |conduit-forwarding|.

                                                            *conduit-forwarding*
SSH forwarding flags (`+L`, `+R`, `+D`, `+W`, `+w`) are only applied to the one
new SSH session that a `:Conduit open` or `:Conduit deploy` call creates (the
interactive terminal, or the deploy-only tunnel), not to Conduit's housekeeping
SSH calls for that connection (the shared ControlMaster's own setup,
connectivity checks, or background `scp`/`rsync` file transfers).

This matters because ssh multiplexes all of these onto a ControlMaster shared
by Conduit profiles with the same host, port, and non-forwarding SSH options.
The master itself is created without any forwarding flags, so it stays a plain,
reusable connection. Forwarding options do not affect either a configured
ControlPath or Conduit's fallback ControlPath; the fallback is keyed by the
non-forwarding options. Each `:Conduit open`/`deploy` call therefore layers its
own tunnel(s) on top of the same authenticated master as a fresh multiplexed
session. Different forwards open independent tunnels rather than conflicting,
and `scp`/`rsync` transfers never see these flags at all.

`+W` requests stdio forwarding, while `+w` requests tunnel-device forwarding;
both receive the same session-only scoping as the port-forwarding flags.

:Conduit exit {connection-key}                              *:Conduit-exit*
        Cleanly close the conduit session for {connection-key}, removing
        remote temporary files and closing the reverse tunnel.

:Conduit disconnect {connection-key}                        *:Conduit-disconnect*
        Forcefully disconnect the named connection.

:Conduit source {connection-key}                            *:Conduit-source*
        Echo and copy the `source` command for the remote conduit RC file to
        the system clipboard.

:Conduit notifications {history|dismiss}                    *:Conduit-notifications*
        Show a history of conduit notifications or dismiss all active
        notifications.

:Conduit stop {get|put|*} {connection-key} {pattern}        *:Conduit-stop*
        Stop ongoing transfer jobs. `pattern` is fuzzily matched against the
        transfer's local and remote file names. Use `*` to stop all matching
        jobs for that connection and transfer type.

        Connection keys are resolved by exact key, by host name, or by a
        unique prefix. When multiple profiles are open for the same host, use
        the full profile key shown by completion or in notifications.

:Conduit socket {connection-key}                            *:Conduit-socket*
        Show the ControlMaster socket path used for {connection-key}.

4. Configuration                                            *conduit-config*

g:conduit_default_split                                     *g:conduit_default_split*
        Default split mode when `lvim` is called without an operation.
        Default: "split"

g:conduit_overwrite_vim                                     *g:conduit_overwrite_vim*
        If true, aliases `vim` to `lvim` in the remote shell.
        Default: true

g:conduit_use_popup                                         *g:conduit_use_popup*
        If true, uses popup menus for file selection (e.g., when `lvim put`
        has multiple fuzzy matches).
        Default: false

g:conduit_put_max_depth                                     *g:conduit_put_max_depth*
        Maximum depth for fuzzy searching local files during `lvim put`.
        Default: 5

g:conduit_verbose                                           *g:conduit_verbose*
        Enable verbose logging of conduit operations.
        Default: false

g:conduit_sep                                               *g:conduit_sep*
        Separator used for internal communication.
        Default: ":::"

g:conduit_fallback_shell                                    *g:conduit_fallback_shell*
        Shell to use on the remote host if not specified in
        `g:conduit_host2shell`.
        Default: "bash"

g:conduit_host2shell                                        *g:conduit_host2shell*
        Dictionary mapping hostnames to remote shell paths.
        Default: {}

g:conduit_default_control_persist                           *g:conduit_default_control_persist*
        Default SSH ControlPersist value for manually managed multiplexing.
        Default: "60m"

g:conduit_host2sshoptions				    *g:conduit_host2sshoptions*
        Dictionary mapping hostnames to custom SSH flags. Flags are passed
        directly to ssh/scp/rsync command lines without validation. Use with
        caution; invalid or unsafe options may cause sessions to fail or
        behave unexpectedly.
        Default: {}

g:conduit_borderchars                                       *g:conduit_borderchars*
        List of 8 characters for popup borders.
        Default: Unicode if `+multi_byte`, otherwise ASCII fallbacks.
        `['─', '│', '─', '│', '╭', '╮', '╯', '╰']` (Unicode)
        `['-', '|', '-', '|', '+', '+', '+', '+']` (ASCII)

g:conduit_success_timeout
	The number of milliseconds to persist success notifications.
        Default: 5000

g:conduit_failure_timeout
	The number of milliseconds to persist failure notifications.
        Default: 5000

g:conduit_connect_timeout			    *g:conduit_connect_timeout*
	Seconds to wait before an ssh connection is timed out, equivalent to
	ssh's `ConnectTimeout`.
        Default: 10

g:conduit_connection_attempts			    *g:conduit_connection_attempts*
	Number of connection attempts before giving up, equivalent to ssh's
	`ConnectionAttempts` option.
        Default: 1

g:conduit_echo_socket				    *g:conduit_echo_socket*
	Use `echom` for the `:Conduit socket` command, rather than displaying
	in a notification.
        Default: false

5. Remote Command (lvim)                                    *conduit-lvim*

The `lvim` command is injected into the remote shell. It supports the
following syntax:

`lvim [op...] {args...}`

Operations:
The `lvim` command accepts one or more operations, followed by filenames or
arguments. Conduit identifies the longest prefix of arguments that match
available operations. Multi-word operations like `vert split` are recognized
automatically without requiring quotes.

Available {op}s are:
    - split
    - sp
    - vsplit
    - vsp
    - vert split
    - vertical split
    - tabe
    - tabedit
    - tabnew
    - tab split
    - tab sp
    - tab vsplit
    - tab vert split
    - tab vertical split
    - tab vsp
    - arga
    - argadd

{op}s can also be modifiers:
    - tab
    - vert
    - vertical
    - hor
    - horizontal
    - lefta
    - leftabove
    - abo
    - aboveleft
    - rightb
    - rightbelow
    - bel
    - belowright
    - to
    - topleft
    - bo
    - botright

Examples:
- `lvim vert split file.txt` — Opens file.txt in a vertical split (no quoting needed)
- `lvim tab split file.txt` — Opens file.txt in a new tab split
- `lvim belowright split file.txt` — Opens file.txt in a below split
- `lvim file.txt` — Opens file.txt using |g:conduit_default_split|
- `lvim -- file.txt` — Explicitly marks file.txt as a filename (useful if a filename matches an operation name)

File Transfer Operations:
- put {local_file} [{remote_file}]: Upload local {local_file} to remote {remote_file}. If `{remote_file}` is omitted, Conduit uses the remote shell's current directory.
- get {remote_file} [{local_file}]: Download remote {remote_file} to local {local_file}. If `{local_file}` is omitted, Conduit uses the local current directory.
- mget {pat}: Download remote files matching {pat}. If `{pat}` is not quoted, it will expand on the server and each individual file will be downloaded.
- mput {pat}: Upload local files matching {pat}. If no destination is provided, Conduit uses the remote shell's current directory. `{pat}` must be quoted, otherwise it will be expanded before by the shell before Conduit can access it.

If no operation is provided, |g:conduit_default_split| is used.

For `put`, if the local path is not found, conduit will attempt a fuzzy
search in the local CWD up to |g:conduit_put_max_depth|.

`Conduit open` and `Conduit deploy` also accept SSH options before the host,
using either their short or long spelling:

`:Conduit open +J user1@host1 user2@host2`

or equivalently:

`:Conduit open ++jump user1@host1 user2@host2`

This is translated to `ssh -J user1@host1 user2@host2` and applies to the
entire Conduit session.

Conduit identifies each connection by a profile key derived from its host,
port, and the effective SSH options used to open it (including any
options set via |g:conduit_host2sshoptions|). Repeating `:Conduit open` for a
host+port+options combination that is already open resolves to the *same*
profile key: Conduit multiplexes through the existing SSH ControlMaster and
just attaches another terminal to it, rather than opening a brand new SSH
connection. This means repeat opens do not require re-authenticating (e.g.
under MFA) since they piggyback on the already-authenticated connection.

A distinct profile is only created when the effective SSH options differ from
an already-open profile to the same host and port (for example, connecting
via a different jump host). In that case Conduit appends a short hash of
those options to the host and port to keep the profile keys unique, for
example `user@host:22-1a2b3c4d5e6f`. Use the resulting key (hashed or not)
with `:Conduit exit`, `:Conduit disconnect`, `:Conduit source`, and
`:Conduit stop`.

6. Notifications                                            *conduit-notifications*

Conduit uses a built-in notifier (based on popup windows) to show progress for
file transfers and status updates. If `+multi_byte` is enabled, it uses
fancy Unicode symbols (✓, ×, →, etc.); otherwise, it falls back to
ASCII equivalents (=, x, ->, etc.).

g:notifier_maxwidth                                         *g:notifier_maxwidth*
        Maximum width of notification popups.
        Default: &columns / 2

g:notifier_overflow                                     *g:notifier_overflow*
        How notifications handle messages wider than `g:notifier_maxwidth`.
        Supported values:
                'wrap'      wrap onto multiple lines
                'truncate'  keep one line and trim with an ellipsis
                'carousel'  keep one line and scroll through the full message
        Default: 'carousel'
        In carousel mode, loading indicators such as spinners and progress bars
        stay fixed while the message text scrolls. Notifications may also be
        given `prefix` and `subprefix` options; these stay fixed before the
        message text, and for spinner/progress notifications they are shown
        after the spinner or progress bar.

NotifyPrefix                                             *NotifyPrefix*
        Highlight group used for notification prefixes.

NotifySubPrefix                                       *NotifySubPrefix*
        Highlight group used for notification sub-prefixes.

NotifyProgressBar                                   *NotifyProgressBar*
        Highlight group used for notification progress bars.

NotifySpinner                                           *NotifySpinner*
        Highlight group used for notification spinners.

NotifyMessage                                           *NotifyMessage*
        Highlight group used for notification message text.

g:notifier_carousel_interval                 *g:notifier_carousel_interval*
        Delay in milliseconds between carousel frames.
        Default: 300

g:notifier_carousel_end_pause               *g:notifier_carousel_end_pause*
        Number of seconds to pause after a complete carousel cycle, while the
        beginning of the message is visible again.
        Default: 0

They automatically adapt to `+multi_byte` support.

You can view the notification history using

`:Conduit notifications history`

or dismiss all active notifications using

`:Conduit notifications dismiss`
